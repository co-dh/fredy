## (A) The Ab-representation / relational-calculus blocker

`effective_regular_additive_is_abelian` is the cleanest goal (plain `∀ … IsNormalSubobject`,
no diamond), so it pins down the irreducible obstruction precisely:

For a monic `m : A ↣ B`, Freyd's proof manufactures a cover `q : B ↠ C` with `m = ker q`
by applying the effectiveness axiom to an **equivalence relation** `E` on `B`.  `E` is the
relation "`b` and `b'` differ by an element of the image of `m`".  Making `E` **reflexive**
requires the additive **subtraction** `B × B → …` arising from the faithful representation
`T : 𝒞 → Ab` of §1.552 — i.e. exactly the Ab-calculus.  The naive categorical candidate
`E = ⟨m,m⟩` is reflexive only if `m` is split epi, which is not given (confirmed in the
earlier analysis and unchanged).  `EffectiveRegular.effective` consumes an equivalence
relation and returns the cover; it cannot be invoked until `E` is built, and `E`'s
reflexivity is the Ab-calculus.  The `five_lemma`/`snake_lemma` and the two remaining
`abelian_iff_*` characterizations bottom out in the same calculus (and, for the diagram
lemmas, the §1.543 capitalization metatheorem that lets a bicartesian Horn sentence be
checked in `Ab`), all owned by concurrent work and not yet importable.

`monic_kernel_of_cokernel` (already proven, not a sorry) shows that once an `ExactCategory`
instance is in hand, normality is immediate — so the entire difficulty is *producing* the
exact / Ab-calculus structure, never consuming it.

## (B) Statement-level instance-diamond defect — `abelian_iff_regular_additive_all_normal`

Newly identified and verified via the LSP (was previously mis-attributed solely to the
Ab-calculus).  Two independent obstructions:

1. **Forward** (`all_normal → Nonempty AbelianCategory`): `AbelianCategory` has a
   multi-parent diamond — `RegularCategory` and `HalfAdditiveCategory` both extend
   `HasTerminal`/`HasBinaryProducts`, and `HasZeroObject` re-extends
   `HasTerminal`/`HasCoterminator`.  The anonymous-constructor assembly
   `{ … with all_normal := h }` is rejected with "field `toHasZeroObject` … has already
   been specified".  Reconciling the parents is real structural work, and the mathematical
   core still needs (A).

2. **Reverse** (`Nonempty AbelianCategory → all_normal`): a genuine *statement* defect, not
   a calculus gap.  `IsNormalSubobject m hm` references `Kernel f`, which depends on the
   ambient `[HasZeroObject 𝒞]` / `[HasEqualizers 𝒞]` — classes carrying chosen *data*.  An
   arbitrary `Nonempty (AbelianCategory 𝒞)` witness has its own such data, so
   `inst.all_normal m hm` has type `@IsNormalSubobject 𝒞 _ inst.toHasZeroObject
   inst.toHasEqualizers …`, not the goal's `@IsNormalSubobject 𝒞 _ inst✝² inst✝¹ …`.  This
   is a real type mismatch with no transport.  A faithful fix would change the statement to
   force the `AbelianCategory` instance to extend the ambient instances; since statements
   are verbatim per task, the sorry is retained.
