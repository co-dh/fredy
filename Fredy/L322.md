# LeetCode 322 (Coin Change): why it can't genuinely use the AoP dynamic-programming theorem

**Punchline.** Coin change *looks* like a dynamic program, but it does **not** fit Bird & de Moor's
Theorem 9.1 (`Fredy.A9_1.dynamic_programming`). This is not a missing-lemma problem — the mismatch is
real and provable. The hand proof in `L322.lean` is the faithful correctness; the theorem cannot
replace it. Here is exactly why.

## The program

`dp coins amount` = fewest coins summing to `amount`, or `none` if impossible. It recurses on the
**amount**: to solve `a`, try every coin `c ≤ a`, recurse on `a − c`, add `1`, and take the minimum.

```
dp coins 0       = some 0
dp coins (a+1)   = min over { c ∈ coins | 1 ≤ c ≤ a+1 } of  (dp coins (a+1−c) + 1)
```

## The theorem it "should" match

`dynamic_programming` (A9_1) says: the DP fixpoint refines "generate every solution, take the min".

```
mu (fun X => A (T°) ≫ powerRel (F.map X ≫ h) ≫ minRel R)      -- the DP recursion
  ⊑  A ((relCata I T)° ≫ relCata I h) ≫ minRel R              -- generate-all-then-min = the spec
```

with `T` the coalgebra (unfold), `h` the algebra (refold), `R` the cost order.

## Every hypothesis actually holds — the instantiation is *legal*

Map coin change onto it at `𝒜 = Rel(Set)`:

- `F X = Unit ⊕ (Nat × X)` (coin sequences), `I = initial` its initial algebra — both exist.
- `T`: unfold `a` into `(coin c ≤ a, sub-amount a−c)`.   `h`: refold by `+1`.   `R = (≤)`.
- `Map h` ✓,  `R ≫ R ⊑ R` ✓ (`≤` transitive),  `MonotonicAlg h R` ✓.

The **spec side even identifies cleanly**: with `H = (relCata I T)° ≫ relCata I h`, you get
`H a n ↔ Achievable coins n a`, and `A H ≫ minRel R` is exactly `coinSpec`. So the right-hand side of
the theorem *is* our specification. The theorem's hypotheses are all satisfiable. Nothing is missing.

## What breaks: the DP body demands *every* branch succeed

The body uses the **Egli–Milner** power relator `powerRel` (`A5_4`). Its meaning on `Rel(Set)`:

> `powerRel g P Q`  ⟺  every `t ∈ P` reaches some `u ∈ Q` with `g t u`.

"Every element of the unfolded set." And `A (T°) a` is the **full** decomposition set — *all* valid
coins `c ≤ a` at once. So the least fixpoint `mu(body) a` is non-empty only when **every** valid coin
leaves a *solvable* sub-amount `a − c`.

But coin change needs only **one** good branch. That is the whole mismatch.

### Concrete counterexample

`coins = {2, 3}`, target `3`:

- `3` is solvable: one `3`-coin, so `dp = some 1`.
- Branch `c = 2` leaves sub-amount `1`, which has **no** valid coin (`2, 3 > 1`), so `mu(body) 1 = ∅`.
- Egli–Milner requires *all* branches productive ⇒ the one dead branch forces `mu(body) 3 = ∅`.

So `dp`-as-a-relation is **not** `⊑ mu(body)`, and `mu(body) ⊑ spec` (the theorem) therefore cannot
carry `dp`'s correctness. Moving to the `Option Nat`/∞ lattice does not help — it only relocates the
same mismatch to the no-valid-coin amounts, where `minRel` over an empty decomposition set produces
*nothing* while the executable produces `none`.

## The general lesson

Theorem 9.1 faithfully models DPs whose coalgebra **unfolds a finite input structure**, so every
branch is productive:

| Fits Theorem 9.1 (recursion over given input) | Does *not* fit (recursion over a value, dead ends) |
| --------------------------------------------- | -------------------------------------------------- |
| edit distance, bracketing, compression        | coin change, unbounded knapsack by weight          |
| every unfold step consumes one input element   | unfold an amount into decompositions; answer comes |
| ⇒ all branches productive                      | from a *single* surviving branch                   |

Coin change recurses on a **value** (the amount) into decompositions with dead ends. It is the second
kind, so `A9_1` does not apply.

It *could* be made Theorem-9.1-faithful only by rewriting it as a **catamorphism over the denomination
list** (process each coin, unbounded-knapsack style), where every branch is structural and productive
— but that is a different program, not the amount-recursion `dp` we have.

## Why this matters for the "LeetCode in allegory" project

This is the sharp answer to "most of Freyd's / Bird–de Moor's theorems aren't used in the solutions."
The AoP calculational theorems apply to a **specific class**: structural recursion over input data,
verified via catamorphism + fusion (see `A6_6_Sort.selection_sort_correct` for the genuine template).
Many problems — value-recursion searches like coin change — are simply not in that class. For those, a
direct induction is the **correct, faithful** proof, not a shortcut worth apologizing for. Forcing a
nominal theorem call that does no real work would be gerrymandering, not "using AoP".
